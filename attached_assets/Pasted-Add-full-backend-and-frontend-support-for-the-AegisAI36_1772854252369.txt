--------------------------
Create:

PART 2 — PLAN LIMITATIONS (CRITICAL)
==================================================

Add a new table: plans
Fields:
- id
- name
- price
- max_agents
- max_logs_per_day
- max_commands_per_day
- allow_network_isolation (boolean)
- allow_process_kill (boolean)
- allow_file_scan (boolean)
- allow_endpoint_download (boolean)
- allow_terminal_access (boolean)
- allow_threat_intel (boolean)
- allow_advanced_analytics (boolean)

Add a new table: usage_tracking
Fields:
- user_id
- date
- agents_registered
- logs_sent
- commands_executed
- terminal_commands_executed

Enforce plan limits:
- When agent registers → check max_agents
- When logs are sent → check max_logs_per_day
- When commands are executed → check max_commands_per_day
- When user tries to isolate network → check allow_network_isolation
- When user tries to kill processes → check allow_process_kill
- When user tries to scan files → check allow_file_scan
- When user tries to download agent installer → check allow_endpoint_download
- When user tries to open terminal or execute terminal commands → check allow_terminal_access
- When user uses threat intelligence features → check allow_threat_intel
- When user uses advanced analytics/AI features → check allow_advanced_analytics

If limit exceeded:
Return error: “Your current plan does not allow this action. Please upgrade.”

PART 5 — BUILT-IN TERMINAL / REMOTE COMMAND SHELL
==================================================

1. FRONTEND — Terminal UI
--------------------------
Route: /admin/endpoints/:agentId/terminal

Features:
- Terminal-like interface:
    • Input box for commands
    • Scrollable output area
- Show warning banner:
    “All commands are logged. Destructive commands are blocked.”
- Disable access if:
    • plan.allow_terminal_access = false
    • user role is not allowed (e.g., only Admin/Responder)

2. BACKEND — Terminal Command Handling
--------------------------------------
Extend agent_commands to support terminal commands:

- command: "terminal_exec"
- params:
    • "command": string (e.g., "ps", "netstat -ano", "dir C:\\Users")

Create endpoint:
POST /api/agent/terminal/execute
- Accepts: agent_id, command
- Validates:
    • User has permission
    • Plan allows terminal access
    • Command is in allowed list (whitelist)
- Creates agent_commands entry with type "terminal_exec"

Allowed commands (example whitelist):
- Windows:
    • "whoami"
    • "ipconfig"
    • "netstat -ano"
    • "tasklist"
    • "dir <path>"
- Linux:
    • "whoami"
    • "ifconfig" or "ip a"
    • "netstat -tunap" or "ss -tunap"
    • "ps aux"
    • "ls <path>"

No destructive commands (no rm, del, format, shutdown, etc.).

3. AGENT SIDE (STUB)

PART 6 — SECURITY REQUIREMENTS
==================================================

- All agent endpoints require agent_id + device_token
- All communication over HTTPS
- Input validation on all APIs
- Rate limiting on agent and terminal endpoints
- Sanitize logs and command results
- Enforce RBAC:
    • Roles: super_admin, admin, analyst, viewer
    • Terminal and destructive actions only for allowed roles
- Never allow:
    • Privilege escalation
    • Arbitrary shell access outside whitelist
    • Silent installation or hidden behavior
